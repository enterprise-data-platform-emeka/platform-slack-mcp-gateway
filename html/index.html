Slack and MCP analytics gateway

Slack becomes a governed analytics entry point.

This repository listens to Slack app mentions and direct messages, converts them into analytics tool calls, sends the question to the existing FastAPI agent, and posts a Slack-native answer with report assets, cost, assumptions, and verification details.

Frontend position

The gateway is Slack plumbing around the existing analytics brain.

Additive surfaceSlack adds a third entry point without replacing Streamlit or the custom HTML dashboard.
No duplicate analyticsSQL guardrails, Claude calls, Athena execution, charting, cost, and audit all remain in the FastAPI agent.
Right level of detailSlack shows a short answer first, then puts cost, assumptions, and verification details in the thread.
Slack runtime sequence

A Slack message becomes an analytics answer and a PDF report.

Thread-aware sessionsThe gateway stores session IDs by Slack thread key and falls back to recent channel session context.
Ordered repliesThe summary posts first, the PDF uploads next, and the cost/assumption footer posts last.
Backend reuseThe gateway calls `/ask` and `/report/pdf`; it never reaches directly into Claude, Athena, or S3.
MCP-style boundary

The Slack listener talks to one analytics tool facade.

Slack adapterHandles app mentions, DMs, channel filtering, thread/session mapping, and Slack formatting.
Tool facadeProvides the stable gateway boundary: ask a data question and build a report.
API clientUses the standard library HTTP client to call the existing analytics agent.
Guardrails and output shape

Slack controls who can ask; the agent controls what can run.

Gateway-level controlsChannel allowlisting and DM filtering prevent unintended Slack traffic from reaching the analytics agent.
Backend-level controlsThe SQL rules remain centralized in the analytics agent, so every frontend follows the same policy.
Thread transparencySlack gets a short answer while technical details stay available in the same conversation thread.
Deployment and secrets

GitHub deploys the gateway and moves Slack tokens into Secrets Manager.

GitHub secrets sourceEnvironment-scoped Slack tokens are validated during deploy before AWS changes happen.
AWS runtime secretThe workflow writes Slack app and bot tokens into Secrets Manager for the ECS task to consume.
Stable Slack identityThe Slack workspace and app are not recreated for every demo; only the AWS gateway runtime is disposable.

Repository mental model

Slack event → tool facade → agent API → threaded reply

The repo is a Slack adapter and gateway. It keeps conversation handling separate from analytics logic.

Why Socket Mode

No public Slack callback URL → ECS connects outbound

For demos and dev, Socket Mode keeps setup simpler while still supporting app mentions and direct messages.
